01
Platform

What Neraium Is

Neraium operates as a read-only infrastructure intelligence layer. It receives telemetry, analyzes sustained deviations, preserves evidence trails, and delivers information to operators. Autonomous control or actuation is intentionally excluded.

CAP-001
Telemetry Ingestion

Receives streams from existing systems without issuing commands. Built to handle varied signal domains with minimal disruption.

CAP-002
Deviation Analysis

Detects sustained multi-signal changes that reveal emerging patterns beyond isolated events or thresholds.

CAP-003
Operator Review

Results are provided for review by licensed personnel, maintaining clear boundaries between analysis and decision-making.

CAP-004
Evidence Records

Maintains structured context of inputs and analysis states to ensure accurate review and future reference.

CAP-005
Infrastructure Focus

Aimed at physical systems where telemetry review and traceability are essential to operational integrity.

CAP-006
Current Stage

Current public proof is a technical foundation intended for demonstration. Production validation awaits domain-specific data.

02
Architecture

System Architecture

A three-stage design—ingest, analyze, review—keeps boundaries clear and avoids overstated capabilities. Each layer is independently responsible for a specific function.

Layer 01
Ingest

Receive and normalize incoming sequences, retaining raw context for later review.

Layer 02
Analyze

Apply multivariate deviation logic and compare against baselines to flag persistent changes.

Layer 03
Review

Deliver context-rich output for interpretation without directly acting on the system.

03
Evidence

Evidence Basis

Current public demonstrations reference the NASA C-MAPSS dataset as an algorithm-development benchmark. This should be clearly distinguished from claims of field performance.

Dataset
NASA C-MAPSS, a public turbofan engine benchmark used for run-to-failure sequence analysis.
Support
Useful for developing and testing multivariate drift logic, baseline comparisons, and scoring methods.
Limit
Does not establish proven performance in water systems or live infrastructure environments on its own.
Framing
Best described as a technical demonstration, not a field-validated solution.
04
Deployment

Deployment Model

An intended design—connect to existing telemetry, remain read-only, validate on domain data—guides the deployment path. It does not imply completed field rollout without evidence.

PLAN-001
Use Existing Telemetry

Designed to integrate with current systems rather than requiring proprietary sensing equipment.

PLAN-002
Preserve Boundaries

Remains read-only, ensuring that operators maintain responsibility for decisions and actions.

PLAN-003
Validate with Domain Data

Next steps involve analysis on infrastructure-specific telemetry to align results with actual systems.
