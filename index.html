01
Platform

What Neraium Is

Neraium is framed as a read-only infrastructure intelligence layer. The platform is intended to ingest telemetry, analyze persistent deviation patterns, preserve reviewable records, and present information for human review. It does not issue control commands or claim autonomous system actuation.

CAP-001
Read-Only Telemetry Ingestion
Designed to receive telemetry from upstream systems without sending control actions back into the operating environment.
CAP-002
Deviation Analysis
Built to identify sustained changes across multiple signals so operators can review emerging patterns rather than isolated events.
CAP-003
Operator Review
Outputs are intended for licensed personnel, facility operators, and other authorized stakeholders who retain decision authority.
CAP-004
Reviewable Records
Structured records of inputs, analysis states, and reviewed outputs are part of the intended system design.
CAP-005
Infrastructure Focus
The target domain is physical infrastructure where telemetry review, traceability, and accountability are operationally important.
CAP-006
Current Stage
Current public proof is best described as a technical and analytical foundation, not a completed field-validated production record.
02
Architecture

Current System Model

The current framing is a three-part read-only flow: ingest, analyze, and review. This keeps system boundaries clear and avoids overstating autonomous capability.

Layer 01
Ingest
Receive source telemetry, normalize time-series inputs, and preserve the underlying data lineage required for review.
→
Layer 02
Analyze
Apply multivariate logic to identify persistent deviation, compare against baseline behavior, and generate reviewable analytical output.
→
Layer 03
Review
Present results for human interpretation and preserve records without claiming automatic control of the underlying system.
03
Evidence

What Is Actually Demonstrated Today

This section is limited to claims that can be stated cleanly. The current analytical basis can reference the NASA C-MAPSS dataset as an algorithm-development benchmark, not as direct field proof for water infrastructure.

Source Dataset
NASA C-MAPSS, a public turbofan engine degradation benchmark commonly used for run-to-failure prognostics and time-series analysis.
What It Supports
Testing of multivariate degradation detection logic, baseline comparison methods, and sequence-based deviation scoring under controlled benchmark conditions.
What It Does Not Support
It does not, by itself, prove performance in water systems, commercial facilities, municipal environments, or live operator workflows.
Correct Framing
Current results are best described as analytical demonstration, prototype validation, and evidence of technical direction. Field validation still requires relevant infrastructure telemetry.
Next Step
Analysis on domain-specific infrastructure telemetry where observed results can be tied directly to the operating environment.
04
Deployment

Planned Deployment Model

This section describes intended operating design. It explains how the platform is meant to be implemented without implying completed field rollout where that has not yet been shown.

PLAN-001
Use Existing Telemetry
The intended model connects to telemetry already present in the environment rather than assuming proprietary sensing hardware.
PLAN-002
Preserve System Boundaries
Read-only observation and analysis only. Operators and authorized teams remain responsible for downstream action and operational decisions.
PLAN-003
Validate on Domain Data
The next factual validation step is analysis on relevant infrastructure telemetry where results tie directly to the operating domain being served.
